# `subject_access_points`

**Type**: Array of strings

---

The `subject_access_points` array in AtoM contains zero-or-more simple subject terms (keywords) as plain strings, used to aid discovery. RiC-O v1.1 does not define a dedicated subject entity in the core; instead, it provides the relation **`rico:hasOrHadSubject`** from a `rico:RecordResource` to a generic `rico:Thing`. To model shared, reusable subjects simply and consistently, we use **SKOS**:

- Create one `skos:Concept` per unique subject term, grouped by a single `skos:ConceptScheme` (e.g., "Subjects").
- Link each record resource to the relevant concept(s) using **`rico:hasOrHadSubject`**.

This yields a flat (non-hierarchical) subject vocabulary, matching AtoM's API output which does not provide hierarchy in responses. Future enhancement can introduce hierarchical SKOS (using `skos:broader`/`skos:narrower`) without changing record-to-subject links.

Notes:
- `rico:hasOrHadSubject` domain is `rico:RecordResource`; range is `rico:Thing`. `skos:Concept` is safely usable as the object.
- Keep it simple: no `skos:Collection` is required; use one `skos:ConceptScheme` and `skos:inScheme`.

### RiC-O / SKOS Entities
1. `rico:RecordSet/Record/RecordPart` (described resource; links out via `rico:hasOrHadSubject`)
2. `skos:Concept` (one per distinct subject term)
3. `skos:ConceptScheme` (single shared scheme, e.g., "Subjects")

### Example

```turtle
# Declare a single shared concept scheme
:SubjectsScheme a skos:ConceptScheme ;
  skos:prefLabel "Subjects"@en .

# Two subject concepts created from the AtoM array
:Subject_architecture a skos:Concept ;
  skos:prefLabel "Subject 1"@en ;
  skos:inScheme :SubjectsScheme .

:Subject_keyword_2 a skos:Concept ;
  skos:prefLabel "Subject keyword 2"@en ;
  skos:inScheme :SubjectsScheme .

# Link a record resource to the shared concepts
:RecordSet_123 a rico:RecordSet ;
  rico:hasOrHadSubject :Subject_architecture , :Subject_keyword_2 .
```

### Notes on hierarchy and future enhancements
- AtoM can maintain hierarchical subjects, but its API response for this field is a flat list. We therefore adopt a non-hierarchical SKOS model now.
- Future development may enrich the vocabulary by adding `skos:broader`/`skos:narrower`, external alignments (`skos:exactMatch`, `skos:closeMatch`), or curated `skos:Collection`s for UI facets—without changing the `rico:hasOrHadSubject` links on records.
